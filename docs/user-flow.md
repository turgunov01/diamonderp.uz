# User Flow (Web ERP + Mobile)

Ниже — “теория” как пользователи проходят ключевые сценарии в проекте: кто что делает, в каком порядке и какие данные для этого нужны.

## Основные сущности

- **ERP пользователь** (`erp_users`): роли `admin|hr|procurement` — заходят в веб-дашборд.
- **Сотрудник** (`customers`): роли `customer|cleaner|manager|supervisor` — в основном работают через mobile API.
- **Здание** (`buildings`): верхний уровень рабочего пространства (контекст).
- **Объект** (`objects`): принадлежит зданию; на объектах ведутся задачи, отчёты, документы.
- **Привязка сотрудника к объектам**: `object_pinned` и `object_positions` (по именам объектов) — влияет на доступ в мобильном приложении.
- **Смена**: `work_shift = day|night` — влияет на расчёт/отображение смены и активности.
- **Пароль и флаг смены**: `password` + `must_change_password` (в mobile API это `mustChangePassword`).

## Веб‑ERP: общий сценарий (admin/hr/procurement)

1. **Вход**: пользователь открывает `/login` и логинится по `email + password`.
2. **Роль → доступ**: middleware проверяет доступ к страницам, а сайдбар показывает только разрешённые разделы.
3. **Выбор здания**: в левом меню выбирается “активное здание” — это контекст для объектов/кадров/задач/отчётов.
4. **(admin) Объекты**:
   - создаёт объекты в выбранном здании;
   - активирует/деактивирует объекты (активный объект используется как “по умолчанию” в некоторых разделах).
5. **(hr/admin) Кадры (сотрудники)**:
   - создаёт сотрудника вручную или через массовую загрузку;
   - назначает `workShift`, зарплату, и при необходимости привязку к объектам (`objectPinned/objectPositions`);
   - при первом создании сотруднику обычно выставляется пароль по умолчанию и `mustChangePassword=true`.
6. **(procurement/admin) Задачи по объектам**:
   - создаёт “to‑do лист” по объекту;
   - выбирает сотрудника (исполнителя) и пункты чек‑листа;
   - дальше сотрудник выполняет это в мобильном приложении.
7. **Документы**:
   - в веб‑части создаются шаблоны договоров/документов и “рассылка” (dispatch) на сотрудников;
   - статус подписания видно по `dispatch/signed` данным.
8. **Отчёты/расходы/прочее**:
   - разделы отчетов и финансов используют выбранные здание/объект и период.
9. **Архив/отключение**:
   - сотрудника можно вывести из активной работы (архив/деактивация), что влияет на доступ и отчётность.

## Mobile: сценарий сотрудника (customer/cleaner)

1. **Первый вход**:
   - сотрудник вводит `login` (телефон или username) и пароль по умолчанию;
   - получает JWT token и данные доступа (`objectIds/objectNames`, смена, `mustChangePassword`).
2. **Задачи**:
   - открывает список задач;
   - выполняет чек‑лист/элементы задачи, отправляет прогресс (и при необходимости фото/комментарии).
3. **Документы**:
   - открывает “Договоры/Документы”;
   - видит назначенные dispatch‑документы (по своему `id` или номеру телефона);
   - подписывает документ (подпись сохраняется в storage, создаётся запись `signed_documents`).
4. **Смена пароля (после подписи)**:
   - если `mustChangePassword=true`, приложение показывает модалку смены пароля;
   - вызывает `POST /api/mobile/auth/change-password` с `currentPassword` и `newPassword`;
   - после успеха `mustChangePassword=false` и модалка больше не должна появляться.
5. **Завершение смены**:
   - при необходимости приложение отмечает окончание работы (finish) и фиксирует активность.

## Mobile: сценарий менеджера (manager)

1. Логинится так же через mobile auth.
2. Получает доступ к объектам (обычно через привязку сотрудника к объектам).
3. В разделе задач может получать задачи на проверку/ревью и менять их статус.

## Что влияет на доступ в mobile

- Для `customer|cleaner` список доступных объектов строится из `object_pinned` + `object_positions`.
- Если сотрудник не привязан к объектам, он всё равно может:
  - войти в приложение;
  - видеть документы, назначенные ему по `recipient_ids` или `recipient_phones`;
  - но список доступных объектов может быть пустым.
